RTOS Kernel

§1

Why it exists

This project builds an RTOS kernel entirely from first principles: task structs and stacks, a cooperative scheduler, preemption via timer interrupt, priority scheduling with the classic inversion bug (and the priority-inheritance fix for it), and blocking synchronization primitives — mutex, semaphore, message queue. Everything runs as a host-native process first: no board, simulator, or cross-compiler needed. docs/design.md documents the internal architecture in full.

§2

How it runs

Context switching uses POSIX primitives (ucontext.h) in place of real hardware registers, and a POSIX interval timer (setitimer + SIGALRM) stands in for a hardware tick interrupt — the host-native equivalent of a SysTick ISR triggering PendSV on a real Cortex-M part.

main() --> scheduler_run() --> [Task A] <--context_switch--> [Task B] <--> [Task C] | SIGALRM (tick) forces preemption

Two build targets, one design

The context-switch code is also ported to real Cortex-M4 assembly and boots under QEMU (primary) and Renode (rtos/arm/) — deliberately a small proof that the design ports to real hardware, not a re-implementation of Phases 2–7 on top of it.

Host-native — src/
  • Context switch via swapcontext() (task.c)
  • Tick source: setitimer + SIGALRM
  • Stack: malloc()'d buffer + ucontext_t
  • Full kernel: priority-aware round robin, sync primitives, sleep, canaries, stack high-water mark, context-switch counter
Cortex-M4 — arm/
  • Context switch via PendSV_Handler (context_switch.s)
  • Tick source: real SysTick
  • Stack: fixed array + hand-built exception frame (kernel_arm.c)
  • Round robin only — a minimal 2-task demo, boots under QEMU/Renode

§5

Test suite

make test builds and runs these, in order — each prints a PASS: ... line and exits 0, or aborts on a failed assert() with the file/line.

  1. test_harness
    Boots the scheduler fresh via scheduler_reset() for three independent cases: round-robin fairness, priority ordering, and confirming a reset really does start clean.
  2. test_preemption
    A non-yielding busy-loop task still gets preempted by the tick timer, and another task still gets to run.
  3. test_priority_inversion
    Asserts inversion occurs without priority inheritance and is resolved with it, using tick-count bounds.
  4. test_scheduling
    4 tasks round-robin fairly under task_yield().
  5. test_stress
    600 short-lived tasks across 30 waves; nothing crashes and canaries stay intact throughout.
  6. test_sync_primitives
    A producer/consumer pair through a queue smaller than the item count — every item arrives exactly once, in order.
  7. test_timing
    task_sleep() wakes a task on schedule, and per-task run/ready tick stats accumulate sensibly.
  8. test_diagnostics
    Stack high-water mark scales with real recursion depth (not just "nonzero"), and the context-switch counter reflects real switches, not just ticks.
§6

Memory & concurrency validation

Validated with AddressSanitizer + UndefinedBehaviorSanitizer rather than the "usual" Valgrind/ThreadSanitizer answer — for reasons, not by default:

Why not Valgrind

Not installed in this project's environment, and no passwordless sudo to add it. AddressSanitizer catches the same leak/overflow/use-after-free class of bug — test_stress.c's 600 create/destroy cycles are the main leak-check target. It doesn't fully support makecontext/swapcontext, by its own warning; no false positives were seen across many runs regardless.

Why not ThreadSanitizer

Every "task" is a cooperative or signal-preempted green thread on a single real OS thread — there's no actual concurrent memory access for TSan to instrument, so a clean run wouldn't prove much. The correctness property that matters — every item arrives exactly once, in order, through a queue too small to avoid blocking — is checked with plain assertions in test_sync_primitives.c instead.

Stack canaries + high-water mark

A guard region is written at the bottom (low-address end) of every task's stack at creation and checked on every dispatch. There's deliberately no "top" guard: makecontext() itself writes into that end of the buffer before the task ever runs once. The rest of the stack is separately poisoned with a distinct byte so task_stack_high_water_mark() (Phase 8) can report real peak usage without being confused with canary corruption.

make clean && make SANITIZE=address,undefined
make test
§7

Field notes — Cortex-M4 port

Two bugs surfaced only under real hardware exception semantics, invisible host-native:

01 · Boot scratch stack undersized

The first PendSV needs somewhere to "save" a nonexistent outgoing context before the first real task is picked. That scratch area was sized to the bare 16-word exception frame — exactly enough on paper, but it silently overflowed into the next static in .bss, corrupting task_sp[1] to 0 before it was ever read. Found by bisecting with a memory-mapped debug print over UART, not by guessing. Fixed by giving it real headroom (64 words).

02 · An unrelated-looking edit reintroduced a hang

Renaming a helper and dropping a diagnostic print in main.c — pure refactoring, no logic change — brought back a PendSV/SysTick pending-but-never-serviced hang that only reproduces under Renode's specific emulation. The exact mechanism was never fully root-caused; what's confirmed is that the verified-working main() structure has to stay as-is.

ROS2 client layer — Micro XRCE-DDS (Option A)verified live

On top of the Cortex-M4 port, xrce/ implements enough of the real Micro XRCE-DDS Client protocol — the same wire protocol actual micro-ROS boards speak over their debug UART — to interoperate with a real, unmodified MicroXRCEAgent binary and publish into a real ROS2 graph. Not a custom protocol reimplemented from imagination: every wire-format detail below is checked against eProsima's reference client source, and verified against the actual agent, logs included. This is the foundation Phase 7 (next tab) builds on.

What's implemented

Verified against a real agent, not just its own tests

Docs vs. source: a real discrepancy

eProsima's published transport docs describe the serial framing's CRC as covering the whole frame, RFC 1662-style. The actual reference client only ever CRCs the payload. Implementing this from the docs would have produced a CRC that silently never matches the real agent's — caught by reading the reference source, not the prose summary.

The real agent's own logs

Running this project's hand-built CREATE_CLIENT / CREATE / WRITE_DATA messages against an unmodified MicroXRCEAgent: its logs show session established, then participant created, topic created, publisher created, datawriter created — each from bytes this project constructed, not eProsima's client. Reproduces identically whether the client is a host process over UDP, or real QEMU-booted Cortex-M4 firmware talking the actual serial framing over an emulated UART — proof the transport works, not just the message construction.

A real bug, traced to its actual cause

ros2 topic echo initially rejected every sample with a payload-size mismatch, on both transports. Reading the agent's own TopicPubSubType::serialize() — not guessing — showed why: its generic dynamic topic type always prepends its own 4-byte CDR header, and this project's client was also including its own, producing a genuine double header. Fix: send just the field bytes, let the agent supply the encapsulation. ros2 topic echo now prints a clean, correct, monotonically increasing sequence. Full byte math in xrce/docs/design.md.

The other direction, driven by the real ROS2 CLI

A subscription (participant/topic/subscriber/datareader + READ_DATA) against the same real agent, driven by a plain ros2 topic pub /cmd std_msgs/msg/Int32 "{data: 99}" --once — nothing from this project on the publishing end. This project's client printed received /cmd data=99. DATA's sample bytes turned out to be header-less too, the mirror image of the WRITE_DATA fix above. Another real bug found the same way: READ_DATA defaulted to a one-shot read (the agent treats an omitted delivery control as max_samples(1), not unlimited) — the first value arrived, the second silently didn't, until fixed to request max_samples = 0xFFFF explicitly.

Same subscription, over real serial into QEMU firmware

ros2 topic pub /setpoint std_msgs/msg/Int32 "{data: 4242}" --once against a real agent produced exactly setpoint updated: 4242 in the QEMU-booted firmware's own UART output — confirmed repeatedly, including a negative value to check sign handling.

When the debugging tool is the bug

Getting there took chasing what looked like a real delivery failure through two rounds — both turned out to be bugs in the debug tooling, not the implementation: a relay pty left in default echo mode was corrupting the very binary traffic it existed to observe, and a log search missed text that really was there, just split across two log lines by the relay's read boundaries. Reconstructing the full byte stream instead of grepping line-by-line found the correct value present from the very first "failing" test. Full account in xrce/docs/design.md.

Fault handling, tested directly

Killed the serial agent process mid-session with the firmware left running untouched, then started a fresh agent on the same device: every entity reappeared automatically via the same periodic re-announcement built for the boot-timing race — no firmware restart, no code change. Confirmed functionally recovered (not just entity-recreated) by running the latency benchmark immediately after and getting real round trips back.

7b

Services and actionsverified live

Needed for anything that isn't fire-and-forget — "get current task list," "trigger a self-test," "run a calibration routine that reports progress and can be cancelled."

Services

REQUESTER/REPLIER entities reuse the existing CREATE/WRITE_DATA/READ_DATA machinery almost entirely unchanged — the one real addition is a 24-byte SampleIdentity prefix (ground-truthed against the agent's own FastDDS middleware source, not the client) that correlates a Replier's reply to its request.

$ ros2 service call /self_test std_srvs/srv/Trigger
response:
std_srvs.srv.Trigger_Response(success=True, message='self-test #1 ok')
A naming gap found only by testing

The reference client's own official example creates a Requester from a bare service_name, letting the library derive DDS topic names. Doing exactly that got the agent to accept the Replier — but ros2 service list never showed it, since FastDDS's own default derivation doesn't match ROS2's DDS-RPC naming convention (rq/rr prefix + Request/Reply suffix on the fully-qualified name). Confirmed by standing up a real rclpy service and comparing. Fixed by setting <request_topic_name>/<reply_topic_name> explicitly.

Actions

Turned out to need no new XRCE protocol work at all: per ROS2's own design docs, an action is just three services (send_goal, cancel_goal, get_result) and two topics (feedback, status) under a naming convention — entirely built from what services above and Phase 4's plain topics already provide. The synthesized wrapper types' exact field layout came from reading the real generated ROS2 headers under /opt/ros/jazzy/include/, not guessed.

$ ros2 action send_goal /fibonacci example_interfaces/action/Fibonacci "{order: 5}" --feedback
Feedback: sequence: [0]
Feedback: sequence: [0, 1]
Feedback: sequence: [0, 1, 1]
Feedback: sequence: [0, 1, 1, 2]
Feedback: sequence: [0, 1, 1, 2, 3]
Result:   sequence: [0, 1, 1, 2, 3]
Goal finished with status: SUCCEEDED
A held request, not a polled one

The first version replied "not ready yet" and dropped a GetResult request if the goal wasn't done, assuming a real client would retry. It doesn't: ros2 action send_goal --feedback sends GetResult exactly once, right after goal acceptance, then just waits — confirmed by it never printing a Result: section at all with the polling version. Fixed by holding the request's identity and replying only once the goal reaches a terminal state.

Cancellation, verified with the real ActionClient

The ros2 action CLI has no cancel_goal subcommand, so cancellation was verified with a small script using the standard rclpy ActionClient directly (the same library the CLI itself uses): sends {order: 20}, waits for 4 feedback messages, calls cancel_goal_async(), gets back a real CancelGoal_Response(return_code=0, ...), then a final status of CANCELED with the sequence frozen at exactly the 4 values already sent.

last_acknown reaching its target is a hard, deterministic proof every sample the reliable stream sent was eventually confirmed delivered — best-effort has no equivalent mechanism at all: whatever the proxy drops on that hop is gone, silently, forever. A real, separate ros2 topic echo subscriber downstream shows the same story with real numbers, consistently more than double for reliable, run after run.

QoS is also wired into the real DDS entity itself via the CREATE XML (<historyQos> inside <topic>, <qos><reliability> as its sibling) — a shape found only by reading the agent's own XML-parser rejection log, since two plausible shapes taken from public FastDDS docs were both rejected first.

A session must keep one address

An early version sent setup directly to the agent and only routed data through the lossy proxy. The agent correlates a session with the UDP source address later traffic arrives from; since the proxy relays through one fixed socket, the agent saw a different address for data than for setup and silently dropped all of it — confirmed by temporary per-packet proxy logging showing every packet forwarded correctly while the agent's own log recorded nothing. Fixed by routing everything through the same proxy.

A resent identical CREATE gets no second reply

The agent's best-effort input stream tracks the last sequence number processed per stream and silently drops an exact-byte resend as a replay. Fixed by rebuilding each retry fresh, so every attempt gets its own sequence number — which surfaced a second real finding: some CREATE-of-an-existing-entity replies come back ALREADY_EXISTS (0x82), not OK_MATCHED as earlier phases had assumed.

A reliable stream must start its own counting at 0

This project's session tracks one sequence counter shared across every stream it uses. Setup traffic on the best-effort stream had already advanced it well past 0 by the time the reliable stream was first used, so its first WRITE_DATA carried a sequence number the agent had never seen a stream start at — its ACKNACK correctly, faithfully asked for messages that were never sent on that stream and never could be. Found by logging the agent's actual ACKNACK values rather than assuming the mechanism was broken.

7d

Priority-aware executorverified under real load

Replaces a naive round-robin spin loop with dispatch driven by the RTOS's real task priorities — a known real limitation of the stock ROS2 executor, and a natural extension of the priority-inheritance work already in the kernel's mutex.

A non-blocking poller task drains the UDP socket and dispatches each sample into the matching topic's queue; two consumer tasks at different task_spawn() priorities measure real dispatch latency. xrce/ stays free of any RTOS dependency by design, so this logic lives in host/, the same place the kernel and the XRCE client already get linked for the QEMU firmware demo.

Under real load from two simultaneous ros2 topic pub -r 20 streams — the low-priority consumer deliberately doing slow, non-yielding CPU work per message:

[low]  round done: mean=39144us  max=124432us

A tight poll loop can starve the agent, not just this process

The poller's first version called task_yield() between polls. Being the highest-priority task, that only ever leaves it READY, never BLOCKED, so it kept getting re-selected — the resulting multi-hundred-thousand-iterations-per-second spin was enough to starve the real agent process of scheduling time on this host: data an unmodified subscribe demo receives instantly at the same moment never arrived at all, confirmed by logging errno=EAGAIN on every poll for as long as a minute. Fixed with task_sleep(1) instead of a bare yield — arguably more realistic anyway, since a real embedded poll loop doesn't spin as fast as the CPU allows either.

Two small, real kernel additions to make this honest

task_stack_high_water_mark()

The whole stack (beyond the existing overflow-guard canary region) gets poisoned with a distinct byte at creation; the accessor scans from the bottom up for the first still-poisoned byte — the deepest point the stack pointer has ever actually reached. Verified with a proportional test, not just "nonzero": a 40-deep recursive task's HWM measurably exceeds a 2-deep one's.

scheduler_switch_count()

A real context-switch counter, deliberately distinct from the existing tick counter — a tick doesn't always cause a switch, and a switch can happen without one (a voluntary yield, or blocking on a sync primitive). Incremented at both real swapcontext() call sites, reset by scheduler_reset() like every other counter.

What isn't tracked — queue drop counts, missed deadlines — is stated as a known gap rather than invented: queue_send() blocks rather than drops by design, and this kernel has no deadline concept yet, only sleep wake times.

Multi-node scalingverified live, 3 boards

N separately-built firmware images, each booted under its own QEMU instance, all appearing as N distinct ROS2 nodes to one real agent process — not N agent processes, and not a custom host-side relay.

The agent already does the hard part

The real agent's multiserial transport mode natively demuxes multiple clients on one process — existing eProsima behavior, not something this project builds. A real agent-CLI gotcha, found by testing

The obvious -D /dev/pts/3 -D /dev/pts/4 (repeated flag) and -D /dev/pts/3,/dev/pts/4 (comma-separated) both fail silently. Reading the agent's own argument parser showed why: multiserial's -D wants one shell argument, space-separated — -D "/dev/pts/3 /dev/pts/4".

Security: HMAC-authenticated linkwrong-key rejection proven live

Real industrial/automotive micro-ROS deployments need authentication on the wire. Every earlier phase's firmware sent completely unauthenticated XRCE-DDS frames — indistinguishable from a bare, unsecured board.

Built from scratch, checked against the spec's own vectors
  • SHA-256 (FIPS 180-4) — matches NIST's published digests, including the million-'a' long-message vector
  • HMAC-SHA256 (RFC 2104) — matches RFC 4231's official test cases, including the >block-size-key path
Why the agent can't be the direct far end

Option A's whole value is zero-modification interop with a real agent. The authenticated hop only exists between firmware and a new host/secure_gateway.c, which verifies/strips it and forwards plain, standard XRCE-DDS on — the same trust-boundary pattern a real automotive gateway ECU uses.

Each frame is wrapped as counter (4B) | truncated 8-byte HMAC tag | inner payload — a strictly-increasing counter defeats replay; the truncated tag (an AUTOSAR-SecOC-style tradeoff) keeps overhead small. Opt-in via a SECURE_LINK=1 build flag — default 0 keeps every earlier phase's direct-to-agent workflow untouched.

Verified live: legitimate traffic, and the actual security property

$ # legitimate key: same /chatter, /setpoint→/pong round trip as always, now authenticated
$ ros2 topic pub /setpoint std_msgs/msg/Int32 "{data: 777}" --once   →   data: 777 on /pong

$ # gateway started with a DIFFERENT key, firmware unchanged:
[uplink] REJECTED frame (bad tag (corrupted in transit, or wrong key)) -- dropped
$ ros2 topic list
/parameter_events   /rosout        # zero firmware topics ever reached the graph

Tamper and replay rejection are covered exhaustively at the unit level (7 cases in test_secure_transport.c) — the same "corruption testing belongs at the unit level" precedent Phase 1 already set. Confidentiality (encryption) and runtime key exchange were explicitly scoped out, stated plainly rather than silently assumed.

Fault injection test suitesystematic, automated, verified live

Earlier phases found real fault-tolerance properties as side effects of chasing other bugs. This phase turns the three fault categories a real deployment needs proof against into a real, repeatable suite — the same discipline the RTOS's own regression harness already applies to the kernel.

Link killed mid-transaction — host/fault_link_kill.sh

4 automated pass/fail assertions: baseline data flowing, firmware survives the agent being SIGKILL'd, a freshly-started agent against the same still-running firmware recovers automatically (the periodic re-announce from Phase 5 doing its actual job under a real kill/restart), and the agent survives the firmware being killed too. All 4 passed live.

Live serial corruption — host/fault_corruption_proxy.py

A byte-corrupting relay on the real QEMU↔agent link, proving live what the unit-level CRC/resync tests already prove. At 0.3% per-byte corruption: full session established, /chatter verified genuinely incrementing (2171 → 2592) while corruption continued. At 2%: never crashed or hung across 90 seconds — just slower, which is what "degrades gracefully" is supposed to mean.

Gazebo: real physics, controlled by this RTOSverified live, 20s of real motion

Every earlier phase proves data flows correctly. This phase changes the question: is anything on the other end actually doing something? rtos/arm/gazebo_demo.c publishes geometry_msgs/Twist on a simulated vehicle's cmd_vel, bridged by a real, unmodified ros_gz_bridge.

A real, previously-latent CDR bug, found by testing against a real subscriber for the first time

xrce/src/cdr.c aligned fields relative to the buffer's absolute start — meaning the 4-byte encapsulation header counted as already-consumed alignment. This had been the documented assumption since Phase 2, and went undetected because no message type with an 8-byte-aligned field immediately after the header had ever been live-verified before. A real RTPS reader rejected every Twist sample: payload size of '56' bytes is larger than the history payload size of '55'.

Ground-truthed the fix with rclpy.serialization.serialize_message() rather than guessing — the header's 4 bytes are followed by zero padding in real Fast-CDR output, not 4. Fixed in the one shared place (align_base, reset by xrce_cdr_write_header()), scoped correctly so session.c — which never calls that function — is provably unaffected (its existing byte-exact CREATE_CLIENT test still passes unchanged).
